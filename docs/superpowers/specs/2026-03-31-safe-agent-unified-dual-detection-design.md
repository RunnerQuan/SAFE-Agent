# SAFE-Agent 联合检测工作台设计

**日期：** 2026-03-31

## 背景

当前 SAFE-Agent 前端将“数据过度暴露检测”和“组合式漏洞检测”拆分到“任务”和“报告”两套页面与心智中，用户需要在任务提交、进度查看和报告阅读之间来回跳转。与此同时，这两类检测的核心输入高度一致：都基于同一份工具 metadata，仅 DOE 检测额外需要 source/sink 配置。

本次重构的目标是把两类检测收敛为“一次输入、一次提交、一次查看结果”的联合检测工作流，同时保留“Skill 可信安全检测”页面及其行为不变。

## 目标

1. 首页三张能力卡片的 icon、文案与外链全部居中显示。
2. “数据过度暴露检测”和“组合式漏洞检测”统一进入同一个新建任务入口。
3. 导航层面保留“首页”“Skill 可信安全检测”“任务”，移除独立“报告”入口。
4. 新建任务页改造成联合检测表单：共享 metadata 输入，高级配置承载 DOE 专属 source/sink。
5. 任务详情页兼任报告页：在同一页面展示执行状态、DOE 结果、组合式漏洞结果与综合结论。
6. “Skill 可信安全检测”页面不做任何内容或行为修改。
7. 报告展示参考 AgentRaft DOE Library 的现有输出风格，但以更清晰的前端报告化布局组织。

## 非目标

1. 不重写 SkillPecker 模块。
2. 不改变两类检测背后的算法实现。
3. 不做与本次联合检测工作流无关的全站视觉重构。

## 信息架构

### 导航

保留以下主导航项：

- 首页 `/`
- Skill 可信安全检测 `/skillpecker`
- 任务 `/scans`

删除独立“报告”导航项。

### 页面职责

- 首页 `/`
  - 展示平台介绍与三类能力入口。
  - DOE 与组合式漏洞两张卡片都跳转到 `/scans/new`。
  - Skill 可信安全检测卡片保持跳转 `/skillpecker`。
  - 顶部主操作按钮改为“新建任务”和“查看任务”。
- 任务列表 `/scans`
  - 作为联合检测任务总览页，展示全部历史任务。
- 新建任务 `/scans/new`
  - 负责一次性提交 metadata 并启动联合检测。
- 任务详情 `/scans/[id]`
  - 负责展示执行中状态与最终联合报告。
- 旧报告页 `/reports`
  - 退场；应跳转到 `/scans` 或在 UI 中不再暴露。
- 旧报告详情 `/reports/[id]`
  - 退场；应尽量跳转到对应扫描详情，否则回退 `/scans`。

## 首页设计

### 卡片对齐

首页底部三张能力卡片内部元素统一居中：

- icon 容器水平居中
- 标题居中
- 描述居中
- CTA 链接与外链图标整体居中

### 行为

- 数据过度暴露检测：跳转 `/scans/new`
- 组合式漏洞检测：跳转 `/scans/new`
- 智能体 Skill 可信安全检测：跳转 `/skillpecker`

## 联合检测输入设计

### 任务输入策略

用户已确认采用“共享主输入 + DOE 高级配置”的方案：

- 通用输入：metadata
- DOE 专属输入：source、sink
- DOE 专属输入位于“高级配置”区块中，默认折叠

### 新建任务页结构

#### 1. 顶部摘要区

- 标题：联合安全检测
- 副标题：强调一次输入同时执行 DOE 与组合式漏洞检测
- 检测项状态胶囊：DOE 检测、组合式漏洞检测
- 简短说明：系统基于同一份 metadata 生成两类结果，并在同一任务详情汇总展示

#### 2. 主输入区

- 任务名称
- metadata 文本输入
- metadata 文件上传（JSON）
- 从历史任务复制时自动回填
- 高级配置折叠区
  - Source 函数
  - Sink 函数
  - 帮助文案，说明仅 DOE 使用

#### 3. 提交前确认区

- 工具数量
- metadata 结构校验结果
- DOE 参数是否已配置
- 主按钮：开始联合检测
- 次按钮：重置

### 检测启用规则

为兼顾可用性与 DOE 精度，采用以下规则：

- 组合式漏洞检测默认启用
- DOE 检测在填写 source 与 sink 后启用
- 未填写 source/sink 时，允许任务提交，但页面应明确提示本次仅运行组合式漏洞检测

若当前后端能力要求两项必须同时执行，可在实现阶段提升为提交前校验；但前端信息结构仍保持“高级配置”的承载方式。

## 任务列表设计

任务列表取代原“任务 + 报告”双列表心智，展示“联合检测任务”。

每条任务至少展示：

- 任务名称
- 任务状态
- 创建时间
- 工具数量
- 本次启用的检测项
- 是否已有可读结果
- 查看详情入口
- 复制配置入口
- 取消任务入口（运行中时）

列表摘要卡可展示：

- 总任务数
- 运行中任务数
- 已产出结果任务数

## 任务详情页设计

任务详情页兼任报告阅读页，是本次重构的核心页面。

### 1. 顶部总览卡

展示：

- 任务名称
- 创建时间
- 当前状态：排队中 / 运行中 / 已完成 / 部分完成 / 失败
- 工具数量
- 本次启用检测项
- 综合风险等级
- 操作：重新运行、导出 JSON、下载 PDF（若导出能力存在）

### 2. 执行进度区

采用双轨进度卡：

- DOE 检测进度
- 组合式漏洞检测进度

每个卡片展示：

- 当前阶段
- 产物数量
- 最近更新时间
- 日志入口
- 错误信息（若失败）

支持“部分完成”状态：当一项成功、一项失败时，成功结果仍可阅读。

### 3. 综合结论区

置于页面中上部，优先于原始明细：

- 高风险链路数
- 疑似过度暴露工具数
- 高风险 source 数
- 建议优先处理项
- 面向用户的结论摘要

### 4. DOE 结果区

参考 DOE Library 的现有结果内容组织，但改为结构化前端布局：

- 风险摘要卡
- 可疑任务列表
- 单条任务展开面板
  - task
  - source
  - sink
  - 暴露字段
  - 判断理由
  - 审计说明 / 原始 judge 文本

### 5. 组合式漏洞结果区

基于 MTAtlas / 组合式漏洞产物构建：

- 顶部统计：候选链数、筛选后链数、高风险 source 数
- 链路卡片列表
  - source tool
  - sink tool / sink 类型
  - chain 展示
  - risk level
  - reason
- 展开态
  - 链路明细
  - 涉及工具摘要
  - 风险解释
  - 原始 JSON 入口

### 6. 原始产物区

面向技术用户保留可折叠原始数据区：

- DOE 原始结果
- MTAtlas / 组合式漏洞原始 JSON 或 Markdown
- 下载入口

## 视觉方向

### 风格原则

- 延续现有首页与平台的浅色玻璃质感
- DOE 结果使用暖色风险语义
- 组合式漏洞结果使用冷色链路语义
- 综合结论使用更强对比的摘要卡突出优先级
- 原始 JSON 默认折叠，优先展示结构化摘要

### 报告阅读体验

优先保证：

1. 普通用户先看到综合结论与重点风险
2. 技术用户可以展开链路、审计说明与原始证据
3. DOE 与组合式漏洞的视觉表达可区分，但总体风险 badge 体系统一

## 数据与兼容策略

### 前端任务模型

联合任务模型应能承载：

- 共享 metadata 输入
- DOE 配置状态
- 两类检测的独立执行状态
- 综合状态
- 联合结果摘要
- 原始结果引用

### 兼容旧报告路由

- `/reports`：跳转 `/scans`
- `/reports/[id]`：优先跳对应 scan 详情；若无法映射，跳 `/scans`

### SkillPecker 隔离要求

本次实现不得修改 `/skillpecker` 的页面内容、交互与数据逻辑。

## 实施边界

### 预期会修改的区域

- 首页页面与样式
- 顶层导航
- scans 列表 / 新建 / 详情相关页面
- scans / reports 相关 API 与类型定义
- DOE Library / MTAtlas 结果读取与前端映射层

### 不应修改的区域

- SkillPecker 页面逻辑
- 无关后端算法
- 与联合检测无关的公共页面

## 风险与注意事项

1. 当前工作区存在未整理改动，实施时必须避免回退用户已有修改。
2. 当前字符编码存在异常文本，重构中需要避免继续扩散乱码内容。
3. 若旧 reports API 已被其他页面依赖，需要以跳转或兼容响应逐步退场，而不是直接删除接口。
4. 若 DOE 与组合式漏洞结果的产物格式不一致，需要在服务层做统一映射，而不是把复杂原始结构直接暴露给页面组件。

## 验收标准

1. 首页三张卡片所有内容视觉上居中对齐。
2. 导航中不再出现独立“报告”。
3. 首页 DOE 与组合式漏洞卡片都进入同一个 `/scans/new`。
4. 用户可在一个表单中提交 metadata，并在高级配置中填写 source/sink。
5. 用户在一个任务详情页中同时看到 DOE 结果与组合式漏洞结果。
6. 独立报告页不再作为主流程入口。
7. Skill 可信安全检测页面保持原样。
